1. TECHNOLOGY STACK
================================================================================

The project uses a monorepo architecture structure with several interconnected
sub-services. The technologies employed include:

*   CORE RUNTIME:
    - Node.js (version >= 18)
    - Express.js (v5.x - Backend APIs)
    - React.js (v19 - Frontends)
    - NestJS (Microservice for asynchronous/isolated email logic)

*   DATABASE & STATE:
    - MySQL / MariaDB (version 10.4+ / MySQL 8+)
    - mysql2 (promise-based connection pooling)
    - Prisma ORM (used specifically within the NestJS email-service)

*   FRONTEND BUILD TOOLS & PORTALS:
    - Vite (Fast development server and bundling)
    - TailwindCSS (Utility-first styling system)
    - React Router v7 (Client-side routing)
    - Lucide React (Icon sets)
    - Axios (HTTP client for API requests)

*   EMAIL & NOTIFICATION PIPELINE:
    - Nodemailer (Primary provider via SMTP)
    - Brevo / Sendinblue (Secondary provider via HTTP API)
    - Failover Management: "failoverEmailService.js" automatically falls back
      to Brevo if the primary SMTP service fails, logging occurrences to
      the `email_delivery_log` database table.

*   CONTAINERIZATION & UTILITIES:
    - Docker & Docker Compose (Multi-container orchestration)
    - Nginx (Production reverse proxy and static asset server)
    - Concurrently (Local orchestration script runner)
    - Multer (Multipart form-data handler for file uploads)

3. DETAILED DATABASE SCHEMA (DB SCHEME)
================================================================================

The database `rsm_db` uses normalized tables to map configurations, users, and
application workflows. Key tables and relations include:

1.  USER MANAGEMENT
    *   `users`: Stores credentials for portals.
        - `id` (INT PK, Auto-increment)
        - `application_id` (VARCHAR(20) UNIQUE)
        - `full_name` (VARCHAR(255))
        - `email` (VARCHAR(255) UNIQUE)
        - `password` (VARCHAR(255) - hashed)
        - `role` (ENUM('student', 'admin'))
        - `session_id` (INT FK to `sessions`)
        - `created_at` (TIMESTAMP)

2.  APPLICATIONS & CANDIDATE PROFILE
    *   `applications`: Main application workbook.
        - `id` (INT PK, Auto-increment)
        - `application_id` (VARCHAR(20) UNIQUE, FK to `users.application_id`)
        - `user_id` (INT, FK to `users.id`)
        - `exam_center_1`, `exam_center_2` (VARCHAR(100))
        - `subject` (VARCHAR(100))
        - `category` (VARCHAR(50) - Part-Time/Full-Time)
        - `working_district` (VARCHAR(100))
        - `dob` (DATE), `nationality`, `religion`, `gender`, `community`
        - `parent_name` (VARCHAR(255))
        - Communication Address: `address_1`, `address_2`, `address_3`, `district`, `state`, `pincode`, `mobile`, `phone`
        - Permanent Address: `perm_same_as_comm` (TINYINT), `perm_address_1`, `perm_address_2`, `perm_address_3`, `perm_state`, `perm_district`, `perm_city`, `perm_pincode`
        - Identity verification: `id_type`, `id_number`
        - Awaiting final semester: `is_awaiting_final_sem` (TINYINT)
        - PG Details: `pg_degree`, `score_type`, `score_value`, `year_of_passing`, `pg_university`
        - Exemption exams: `qualified_exams` (JSON)
        - Status: `status` (ENUM: 'Draft', 'Submitted', 'Under Review', 'Approved', 'Rejected')
        - Payment indicators: `payment_status` (ENUM: 'Unpaid', 'Paid', 'Failed'), `payment_reference`, `payment_date`
        - Result verification: `entrance_mark` (DECIMAL), `interview_mark` (DECIMAL), `qualification_status` (ENUM: 'Pending', 'Qualified', 'Direct Qualified', 'Failed', 'Absent'), `admission_approved` (TINYINT), `remarks` (TEXT)

4. CORE BUSINESS LOGICS & WORKFLOWS
================================================================================

1.  ONE-ACTIVE-SESSION CONSTRAINT
    The admin dashboard allows only one active academic session at any given time.
    All users (students, supervisors) registering or updating files are mapped
    automatically to the currently active session.

2.  THE ELIGIBILITY & EXEMPTION ENGINE (DIRECT PASS)
    - During registration/draft, students input national-level qualifications
      (NET, SET, JRF, SLET).
    - If a valid exemptive qualification is provided, the backend applies the
      `qualification_rules` mapping and flags the student application as
      `entrance_exam_status = 'Exempted'` and `direct_pass_status = 'DirectPass'`.
    - These candidates skip the entrance exam entirely. Upon application approval
      and fee payment, they advance straight to the Counselling phase.

3.  HALL TICKET PIPELINE
    - Only candidates who are NOT marked as 'Exempted' (i.e. those required to
      take the written entrance exam) and whose applications are marked
      `status = 'Approved'` are permitted to generate and download hall tickets.
    - Hall tickets contain dates, mapped exam centers (`exam_center_1`,
      `exam_center_2`), and seat designations.

4.  EXAM ATTENDANCE & MARK PROCESSING
    - Post-examination, admins can upload a bulk attendance/mark Excel sheet.
    - The backend processes files via asynchronous workers, logging rows to
      `attendance_upload_logs`.
    - Marks are checked against `entrance_settings` (e.g., standard 50% passing).
      Admin publishes results by locking and updating `result_published_at`.

5.  PREFERENCE-BASED COUNSELLING ALLOCATION
    - Qualified/Direct Qualified candidates are invited to counselling.
    - Candidates pick a set of choices matching their research topic,
      assigning priority ranks to combinations of research centers and
      supervisors (up to a configured threshold).
    - Supervisors' seat allocations check real-time availability using vacancy
      variables (`max_candidates` - `current_vacancy`).

5. SYSTEM SECURITY & DEFENSIVE MEASURES
================================================================================

The application implements enterprise-level security protocols at multiple
layers of the stack:

1.  PASSWORD PROTECTIONS
    - Passwords are encrypted on signup/reset using `bcrypt` (or `bcryptjs`) with
      a work factor / salt rounds of 10. Plain text passwords are never stored.

2.  MUTUALLY ISOLATED SESSION AUTHENTICATION (JWT)
    - Tokens are issued upon login with payload fields (id, email, role, session).
    - Authentication is managed via separate, distinct environment variables
      (`STUDENT_JWT_SECRET`, `ADMIN_JWT_SECRET`, `SUPERVISOR_JWT_SECRET`,
      `CENTER_JWT_SECRET`).
    - Attempting to use a Student token to gain access to Admin APIs will fail
      since the signature validation checks fail under differing secrets.

3.  API LAYER DEFENSES (EXPRESS MIDDLEWARE)
    - HELMET: Disables header signatures like `X-Powered-By` to prevent tech
      stack fingerprinting and restricts HTTP protocol vulnerabilities.
    - CORS: Rejects connections from unauthorized domains. Whitelists local ports
      (5173, 5174, 5175, 5176) and production Netlify instances.
    - RATE LIMITING: `express-rate-limit` monitors requests to authenticate (`/api/auth`)
      and general routes (`/api/*`), throttling IPs that exceed thresholds
      (e.g., max 20 auth calls within 15 minutes in production).

4.  SQL INJECTION PREVENTION
    - Raw database commands are parameterized. Variables are passed as separate
      binding arrays rather than string concatenation, prompting the database
      driver (`mysql2`) to execute them securely as prepared statements.

5.  FILE UPLOAD VETTING
    - File upload endpoints pass through Multer middleware, validating allowed
      types (configured globally in `file_upload_settings`) and size restrictions
      (typically 1MB for certs, 200KB for photos).
    - Uploaded files are renamed using randomized hashes and timestamp keys to
      prevent path traversal exploits.
